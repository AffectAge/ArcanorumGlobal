# Standarts проекта Ages3

Документ фиксирует стандарты разработки интерфейса, игровых механик и производительности.
Цель: предсказуемое качество, масштабируемость и отсутствие деградации при росте данных/онлайна.

## 1. Базовые принципы
- Сервер authoritative: все критичные решения и валидации только на сервере.
- Клиент не является источником истины: только отображение + отправка команд.
- Детерминизм резолва: одинаковый вход -> одинаковый результат.
- Игровые механики должны быть хорошо оптимизированы: без лишних вычислений, с учетом нагрузки и масштабирования.
- Изменения должны быть измеримы: метрики/тайминги/размеры payload до и после.
- Любая оптимизация не должна ломать контракт протокола и текущий UX.

## 2. Стандарты интерфейса (UI/UX)
- Mobile + desktop first-class: никакой «только десктоп» логики.
- Адаптивность: ключевые экраны корректны в диапазоне 360px-1920px+.
- Доступность:
  - интерактивные элементы доступны с клавиатуры;
  - видимый focus-state;
  - контраст текста и фона не ниже AA.
- Состояния UI обязательны: `loading`, `empty`, `error`, `success`.
- Блокирующие overlay только при реальной блокирующей операции, подтвержденной сервером.
- Не допускать ложных «вечных загрузок»: на критичных ожиданиях должен быть timeout/escape path.
- Визуальная консистентность:
  - единая система отступов и размеров;
  - единые токены цвета/типографики;
  - единый паттерн модалок/панелей/уведомлений.

## 3. Стандарты игровых механик
- Новая механика должна быть формализована:
  - входные данные;
  - ограничения;
  - стоимость/награда;
  - условия конфликта;
  - порядок резолва.
- Проверки валидности команд выполняются в момент приема команды и в момент резолва.
- Все конфликтные сценарии обязаны иметь явный код ошибки и понятное сообщение.
- Механики не должны требовать полного сканирования мира в hot-path:
  - использовать инкрементальные индексы;
  - поддерживать их консистентность при каждой мутации.
- Любые фоновые/периодические операции не должны блокировать обработку ходов.

## 4. Производительность и масштабирование
- Требование к алгоритмам:
  - hot-path: избегать `O(N)` по всему миру на каждое действие игрока;
  - предпочтение `O(1)`/`O(logN)` через индексы/кэш.
- Обязательные практики:
  - дельты вместо full-sync;
  - версия состояния + ACK/REPLAY;
  - fallback на snapshot-resync;
  - минимизация payload (короткие ключи/маски).
- Запрещено в hot-path без обоснования:
  - глубокие `JSON.stringify` сравнения на больших структурах;
  - повторные полные обходы `orders/worldBase` на каждую команду;
  - тяжелые sync-операции I/O.
- Кэш:
  - TTL-кэш только для частых read-path;
  - обязательная инвалидация на мутациях.
- Ресурсные бюджеты (ориентиры):
  - средний `WORLD_DELTA` должен стремиться к минимальному размеру относительно baseline;
  - p95 времени `resolveTurn` должен контролироваться метриками;
  - клиентские ререндеры тяжелых компонентов должны быть селективными.

## 5. Стандарты данных и API/WS
- Контракты типов в `packages/shared` являются источником истины.
- Любое изменение протокола:
  - обновить типы;
  - обновить клиент и сервер в одном этапе;
  - зафиксировать миграционные последствия.
- Версионирование состояния обязательно для последовательного применения дельт.
- Ошибки API/WS должны иметь стабильные `code` и предсказуемый формат.

## 6. Тестирование и валидация
- Для критичных контрактов и поведения обязательны тесты:
  - валидация приказов;
  - резолв конфликтов;
  - применение/реплей дельт;
  - инкрементальные индексы и их консистентность.
- Минимум проверки перед merge:
  - typecheck workspace;
  - сценарии ручной проверки по измененному флоу;
  - отсутствие регрессий по текущему gameplay циклу.

## 7. Наблюдаемость и диагностика
- На сервере должны быть метрики:
  - размеры WS-пакетов;
  - глубина replay/history;
  - длительность ключевых фаз резолва.
- Для метрик обязательны endpoint'ы просмотра и reset (для локальной диагностики).
- Любая оптимизация без метрик считается гипотезой, а не подтвержденным улучшением.

## 8. Стандарты кода
- Единый стиль проекта и best practices выбранного стека.
- DRY: избегать дублирования, выносить повторяющуюся логику в абстракции.
- Явные имена переменных/функций, минимизировать «магические» значения.
- Без неоговоренных `TODO`/заглушек в прод-коде.
- Комментарии только там, где логика неочевидна.

## 9. Процессные правила (обязательные)
- Всегда вести `AI_NOTES.md` по этапу в формате:
  - что сделано;
  - почему;
  - риски/ограничения;
  - как проверить (с конкретными шагами «как сделать»).
- Обновлять `README.md`, когда меняется поведение/архитектура/запуск/контракты.
- Вести `CHANGELOG.md` по версиям, на русском языке.
- Не делать скрытых допущений: если есть неопределенность, фиксировать варианты и ограничения явно.

## 10. Чек-лист для любой новой механики
1. Определены правила и конфликтные кейсы.
2. Есть серверная валидация и коды ошибок.
3. Нет полного скана в hot-path, есть индекс/кэш при необходимости.
4. Обновлены типы shared + клиент + сервер.
5. Есть метрики влияния (CPU/latency/payload).
6. Обновлены `AI_NOTES.md`, `README.md` (если нужно), `CHANGELOG.md`.

## 11. Правило применения
- При реализации новых механик и оптимизаций следовать этому документу по умолчанию.
- Отклонения от стандартов допустимы только при явной фиксации причины и trade-off в `AI_NOTES.md`.
