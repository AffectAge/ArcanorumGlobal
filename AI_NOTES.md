# Изменения — Этап 23: админ-управление населением (генерация / очистка / редактирование)

## Что сделано
- На сервере добавлены админ-эндпоинты:
  - `POST /admin/population/generate` — генерация населения для `province|country|world` со стратегией `random|custom`,
  - `POST /admin/population/clear` — очистка населения для `province|country|world`,
  - `PATCH /admin/population/provinces/:provinceId` — ручное редактирование населения одной провинции.
- `GET /admin/provinces` и `PATCH /admin/provinces/:provinceId` расширены полем `population`.
- Добавлены клиентские API-методы в `apps/client/src/lib/api.ts` для новых population-операций.
- В `AdminPanel` добавлена новая категория `Управление населением`:
  - выбор scope и стратегии генерации,
  - JSON-редактор пропорций (для custom-режима),
  - очистка населения по scope,
  - отдельный блок ручного редактирования населения провинции.

## Почему так
- Требование пользователя: централизованное управление населением в админке на уровне провинции/страны/мира.
- Scope + strategy покрывают массовые операции и точечную правку без отдельного сложного конструктора форм.

## Риски / ограничения
- Для custom-режима используются JSON-поля; ошибочный JSON блокирует операцию.
- Очистка реализована через `populationTotal = 0`; распределения сохраняются как metadata для будущей ручной генерации/правки.

## Как проверить
1) Выполни `npm run typecheck -ws`.
- Ожидаемо: ошибок нет.

2) В админ-панели открой `Управление населением`:
- Прогони `Сгенерировать население` для scope `Провинция`, затем `Страна`, затем `Весь мир`.
- Ожидаемо: приходит успех с количеством обновленных провинций.

3) Проверь `Очистить население` для разных scope.
- Ожидаемо: у целевых провинций `populationTotal` становится `0`.

4) Отредактируй одну провинцию в блоке ручной правки и сохрани.
- Ожидаемо: значения населения провинции сохраняются и видны после перезагрузки списка.

# Изменения — Этап 19: MVP-механика населения по провинциям

## Что сделано
- В `packages/shared` расширен контракт мира:
  - `WorldBase.provincePopulationByProvince`,
  - `ProvincePopulation` (`populationTotal`, `culturePct`, `ideologyPct`, `religionPct`, `racePct`, `professionPct`),
  - новая секция дельт `WORLD_DELTA.u`,
  - новый бит маски `WORLD_DELTA_MASK.provincePopulationByProvince`.
- На сервере добавлена нормализация/инициализация населения:
  - проценты по каждой категории нормализуются до суммы `100`,
  - для старых сейвов без населения выполняется автоматическое заполнение по всем ADM1-провинциям,
  - значения детерминированы (через стабильный hash), чтобы избежать nondeterministic резолва.
- `resolveTurn` теперь обновляет население каждой провинции на каждом ходе:
  - `populationTotal` меняется по формуле рождаемость/смертность,
  - процентные карты сохраняются без автоматических сдвигов (будут управляться отдельными механиками).
- Клиентский Zustand-store (`applyWorldDelta`) научен применять новую секцию `u`.

## Почему так
- Пользовательская цель: добавить легкий MVP населения без POP-системы и без сложных сущностей.
- Агрегаты на провинцию дают минимальную цену внедрения и совместимы с текущей архитектурой `WORLD_DELTA`.
- Отдельный delta-bit для населения не ломает существующие контракты и не требует full-sync.

## Риски / ограничения
- Обновление населения сейчас проходит по всем ADM1-провинциям в `resolveTurn` (полный проход по карте).
- При очень большом количестве провинций потребуется переход на инкрементальный апдейт (только touched/active регионы) или фоновые батчи.
- На этом этапе нет UI-редактора населения и нет отдельной админ-валидации payload для ручной правки population-полей.

## Как проверить
1) Типизация:
- Выполни `npm run typecheck -ws`.
- Ожидаемо: ошибок нет.

2) Полный snapshot:
- Запусти `npm run dev`.
- Авторизуйся и проверь `AUTH_OK.worldBase.provincePopulationByProvince`.
- Ожидаемо: у провинции есть `populationTotal` и 5 процентных карт, каждая суммируется до 100.

3) Дельта населения:
- Заверши ход (`REQUEST_RESOLVE` / авто-таймер).
- Ожидаемо: приходит `WORLD_DELTA` с битом `provincePopulationByProvince` и секцией `u`, значения населения меняются.

# Изменения — Этап 18: убрать full-snapshot `cloneWorldBaseSnapshot` из hot-path через partial snapshot/dirty sections

## Что сделано
- Удален путь полного snapshot мира (`structuredClone({...worldBase})`) перед расчетом дельт.
- Добавлен новый partial snapshot-механизм:
  - `cloneWorldBaseSectionSnapshot(mask)` — клонирует только нужные ветки по маске,
  - `toWorldBaseForDeltaDiff(...)` — собирает prev-view для diff из snapshot + текущего состояния,
  - `broadcastWorldDeltaFromSectionSnapshot(...)` — отправка дельт из section snapshot.
- `resolveTurn` переведен на section snapshot с маской:
  - `resourcesByCountry`,
  - `provinceOwner`,
  - `colonyProgressByProvince`.
- Все hot-path endpoint'ы, где раньше брали full snapshot, переведены на целевые маски:
  - `/country/colonization/start` -> `colonyProgressByProvince`,
  - `/country/colonization/cancel` -> `colonyProgressByProvince`,
  - `/country/province-rename` -> `resourcesByCountry + provinceNameById`,
  - `/admin/provinces/:provinceId` -> `resourcesByCountry + provinceOwner + colonyProgressByProvince + provinceColonizationByProvince`,
  - `/admin/provinces/recalculate-auto-costs` -> `provinceColonizationByProvince`,
  - `/admin/countries/:countryId` delete -> `resourcesByCountry + provinceOwner + colonyProgressByProvince`,
  - `/admin/registrations/:countryId/review` (reject) -> `resourcesByCountry + provinceOwner + colonyProgressByProvince`,
  - `/admin/game-settings` (колонизационный блок) -> `provinceColonizationByProvince`.

## Почему так
- Full snapshot копировал весь `worldBase` даже когда менялась одна ветка, что давало лишнюю CPU/GC нагрузку при росте данных и онлайна.
- Partial snapshot режет объем копирования до реально изменяемых секций, сохраняя текущий протокол и механику diff.

## Риски / ограничения
- Корректность зависит от правильности `dirty mask` в каждом месте мутаций.
- Если в будущем добавить новую мутацию и забыть включить нужный бит в маску, можно потерять часть дельты.
- Нужна дисциплина: маска должна соответствовать фактическим изменениям состояния.

## Как проверить
1) Сборка/типизация (как сделать):
- Выполни `npm run typecheck -ws`.
- Ожидаемо: без ошибок.

2) Проверка ключевых сценариев дельт (как сделать):
- Запусти `npm run dev`.
- Выполни:
  - старт/отмена колонизации,
  - переименование провинции,
  - админ-изменение провинции (owner/cost/disable),
  - удаление страны.
- Ожидаемо: `WORLD_DELTA` приходит с корректными секциями (`c/o/n/p/z`) без регрессий поведения.

3) Проверка резолва хода (как сделать):
- Отправь набор BUILD/COLONIZE и завершите ход.
- Ожидаемо: корректные списания/начисления ресурсов, захват провинций и `rejectedOrders`.

4) Проверка производительности (как сделать):
- На локальном профиле сравни время резолва/нагрузку GC до и после на одинаковом сценарии.
- Ожидаемо: снижение стоимости snapshot-фазы при частых мутациях малой части `worldBase`.
