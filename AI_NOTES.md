# Изменения — Этап 13: фикс бесконечного overlay "Идет обработка хода" при входе

## Что сделано
- В `App.tsx` изменен авто-триггер резолва по таймеру:
  - убран локальный перевод overlay в `processing` без подтверждения сервера;
  - вместо этого отправляется `REQUEST_RESOLVE`.
- Добавлена защита от повторного авто-запроса резолва в рамках одного хода:
  - `autoResolveRequestedTurnRef`.
- При `AUTH_OK` overlay принудительно сбрасывается в `idle`.
- При logout overlay также сбрасывается в `idle`.
- Обновлен `CHANGELOG.md`.

## Почему так
- Причина зависания: клиент мог сам включить `processing` (по таймеру), но серверный резолв мог не стартовать в тот же момент, из-за чего экран блокировки оставался надолго.
- Overlay должен отражать фактическое состояние сервера (`TURN_RESOLVE_STARTED`/`WORLD_DELTA`), а не только локальный таймер.

## Риски / ограничения
- В очень редком случае задержек сети может быть небольшая пауза между отправкой `REQUEST_RESOLVE` и показом overlay (до прихода server event).
- Если сервер не примет `REQUEST_RESOLVE`, overlay больше не залипнет, но и автоматический переход хода не произойдет до следующего валидного триггера.

## Как проверить
1) Сборка/типизация (как сделать):
- Выполни `npm run typecheck -ws`.
- Ожидаемо: ошибок нет.

2) Воспроизведение прошлого бага (как сделать):
- Запусти `npm run dev`.
- Включи таймер ходов и зайди в игру в момент, когда ход уже должен был завершиться.
- Ожидаемо: нет бесконечного локального `processing` сразу после входа.

3) Проверка авто-резолва (как сделать):
- Дождиcь конца таймера хода.
- Ожидаемо: клиент отправляет `REQUEST_RESOLVE`, и overlay появляется только после `TURN_RESOLVE_STARTED`, затем закрывается после `WORLD_DELTA`.

4) Проверка logout/login (как сделать):
- Во время/после резолва выйди из аккаунта и снова войди.
- Ожидаемо: при новом входе overlay стартует с `idle` и не наследует старое зависшее состояние.
