# Изменения — Этап: resume-auth без полного snapshot при доступном replay

## Что сделано
- Расширен WS `AUTH`: клиент теперь передает `lastKnownWorldStateVersion`, если локальный `worldBase` уже есть.
- На сервере в обработчике `AUTH` добавлен resume-path:
  - если для `lastKnownWorldStateVersion` доступен непрерывный replay до текущей версии, сервер отправляет `AUTH_OK` **без `worldBase`** и сразу догоняющие `WORLD_DELTA`;
  - если replay недоступен, поведение прежнее: `AUTH_OK` с полным `worldBase`.
- На клиенте `AUTH_OK` теперь поддерживает оба режима:
  - snapshot bootstrap (`worldBase` присутствует),
  - resume bootstrap (`worldBase` отсутствует) — используется локальное состояние; при его отсутствии выполняется `snapshot`-ресинк.

## Почему так
- Это убирает лишнюю передачу полного состояния на коротких reconnect-сценариях и снижает сетевую нагрузку.
- Сохранен безопасный fallback на старый full-state поток, если replay-окно не покрывает нужную версию.

## Риски / ограничения
- Для resume важна корректность replay-истории; при её разрыве сервер корректно вернется к full-state bootstrap.
- `AUTH_OK` теперь опционально содержит `worldBase`, поэтому новые клиенты должны обрабатывать оба сценария.

## Как проверить
1) Подключиться, получить несколько `WORLD_DELTA`, затем разорвать и восстановить WS.
2) Убедиться, что в `AUTH` ушел `lastKnownWorldStateVersion`.
3) Проверить, что при доступном replay сервер не отправляет полный `worldBase`, а шлет `AUTH_OK` + replay-дельты.
4) Искусственно сузить/сломать replay-окно и проверить fallback на `AUTH_OK` с `worldBase`.

# Изменения — Этап: идемпотентная обработка дельт на клиенте

## Что сделано
- Обновлена обработка `WORLD_DELTA` в `apps/client/src/App.tsx`:
  - если пришла дельта с `worldStateVersion <= currentWorldStateVersion`, клиент больше не запрашивает replay,
  - вместо этого отправляется ACK текущей версии и сообщение игнорируется как уже применённое/устаревшее.
- Replay запрашивается только при реальной дырке (`msg.worldStateVersion > currentWorldStateVersion + 1`).

## Почему так
- Раньше любое отклонение от `+1` воспринималось как рассинхрон и могло вызывать лишние replay-запросы даже на дубликаты.
- Новое поведение делает применение дельт идемпотентным и снижает сетевой шум при повторной доставке сообщений.

## Риски / ограничения
- Если сервер начнёт отправлять неконсистентные устаревшие дельты массово, клиент будет их игнорировать (что ожидаемо), но корневая причина останется на серверной стороне.
- Протокол ACK пока остаётся одноточечным (`worldStateVersion`), range ACK не внедрялся на этом этапе.

## Как проверить
1) Подключиться к игре и дождаться получения `WORLD_DELTA`.
2) Повторно отправить клиенту ту же дельту (или дельту с меньшей версией).
3) Проверить, что:
   - replay-запрос не отправляется,
   - состояние мира не применяется повторно,
   - отправляется ACK текущей версии.
4) Отправить дельту с пропуском версии (`current+2`) и убедиться, что replay-запрос отправляется как раньше.

# Изменения — Этап: roadmap оптимизации state-sync (client/server)

## Что сделано
- Подготовлен отдельный план `STATE_SYNC_OPTIMIZATION_PLAN.md` с phased roadmap по оптимизации обмена состоянием между клиентом и сервером.
- План опирается на текущую архитектуру `WORLD_DELTA` + ACK/REPLAY и развивает её в сторону стандартных практик: envelope/versioning, range ACK, идемпотентность, coalescing, тонкие entity-дельты, SLO/метрики.

## Почему так
- Запрошена «как можно лучшая» оптимизация state-обмена в принятых стандартах.
- Отдельный roadmap-документ удобнее для согласования с командой и внедрения по этапам без рискованного big-bang рефакторинга.

## Риски / ограничения
- Документ концептуальный: код протокола пока не менялся.
- Для части предложений (например, range ACK и entity-дельты) потребуется синхронное изменение shared/server/client контрактов.

## Как проверить
1) Открыть `STATE_SYNC_OPTIMIZATION_PLAN.md` и пройтись по разделам P0/P1/P2.
2) Сопоставить предложения с текущим pipeline (`WORLD_DELTA`, `WORLD_DELTA_ACK`, `WORLD_DELTA_REPLAY_REQUEST`, `/world/snapshot`).
3) Утвердить порядок rollout и включить задачи в спринт.

# Изменения — Этап: добыча ресурсов провинции зданиями

## Что сделано
- Добавлена механика добычи из провинциальных залежей в серверном экономическом тике (`resolveBuildingsTurn`):
  - здание может добывать ресурс из `provinceResourceDepositsByProvince`,
  - добытый объем зависит от `productivity` и базового `extractionAmountPerTurn`,
  - при `extractionRequiresDeposit=true` залежь уменьшается на фактически добытый объем.
- Добыча интегрирована с текущей экономикой зданий:
  - ресурс кладется в `instance.warehouseByGoodId`,
  - учитывается в `lastProductionByGoodId`,
  - добавлен отдельный след `lastExtractionByGoodId` для диагностики.
- Обновлены типы и сериализация/нормализация инстансов зданий:
  - `packages/shared/src/index.ts`: `BuildingInstance.lastExtractionByGoodId`.
  - `apps/server/src/index.ts`: normalize/compare/reset для нового поля.
- Расширены настройки контент-записей зданий (сервер + клиент API):
  - `extractionGoodId`,
  - `extractionAmountPerTurn`,
  - `extractionRequiresDeposit`.
- В `ContentPanel` (админка контента) добавлен UI-блок настройки добычи для зданий.
- README обновлен описанием новой механики.

## Почему так
- Требование: дать зданиям возможность использовать залежи ресурсов провинции как источник добычи.
- Реализация через контент-поля здания минимально инвазивна: не ломает существующую торговлю/склады и использует уже существующие данные по залежам.

## Риски / ограничения
- Добыча сейчас работает на уровне инстанса здания и прямого доступа к залежам провинции; нет отдельной системы лицензий/квот.
- `infraCoverage` в расчете здания пока фиксирован как `1` (как и было), поэтому ограничение добычи идет через продуктивность и наличие залежей, а не отдельный cap инфраструктуры добычи.
- Баланс добычи зависит от ручной настройки `extractionAmountPerTurn` в контенте зданий.

## Как проверить
1) Типизация:
- Выполнить `npm run typecheck -ws`.
- Ожидаемо: ошибок нет.

2) Настройка здания:
- В `ContentPanel` открыть категорию `Здания` -> `Экономика и производство`.
- Задать:
  - `Добываемый товар` (например товар с discoverable-настройками),
  - `Объем/ход > 0`,
  - `Требует залежь = Да`.
- Сохранить.

3) Игровой сценарий:
- Убедиться, что в провинции есть залежь по выбранному товару.
- Завершить ход.
- Проверить, что:
  - в инстансе здания вырос `warehouseByGoodId[goodId]`,
  - в провинции уменьшился `provinceResourceDepositsByProvince[provinceId][goodId].amount`,
  - в телеметрии здания есть `lastExtractionByGoodId`.

4) Проверка совместимости:
- Убедиться, что остальная экономика (покупка входов/продажа/зарплаты/неактивность) работает как раньше.
